## 第3章 关键修复详解

### 3.1 DEBT-BYTE-001: 字节序自适应层（P1）

**问题**: v2.5使用LE字节序，v2.6使用BE字节序，升级导致数据不兼容

**方案**: 三重检测算法 + 兼容读取层
- **魔数匹配**（置信度95%）：直接匹配LE/BE魔数
- **版本验证**（置信度75%）：验证版本号合理性
- **启发式检测**（置信度50-70%）：基于数据分布特征

**核心API**:
```typescript
// 自动检测
const result = ByteOrderDetector.detect(buffer);
// result.byteOrder: 'LE' | 'BE' | 'UNKNOWN'
// result.confidence: 0.5~0.95

// 兼容读取
const reader = new HctxReader();
const data = reader.parse(buffer);  // 自动识别字节序
```

**验证结果**: BYTE-001~008 全部通过，检测性能4ms/1000次

### 3.2 DEBT-BYTE-002: 全局LE操作清理（P3）

**问题**: src/目录存在LE操作残留，字节序规范不一致

**方案**: 
- 全目录扫描替换（6个文件，37处LE→BE）
- 自动化审计脚本：`npm run audit:byteorder`
- CI集成：每次提交自动检查

**例外保留**:
- `src/legacy/`：专门用于v1.x文件读取（DEBT-LEGACY-001）
- `byte-order-detector.ts`：检测逻辑需要LE参考

**验证结果**: BYTE-009~014 全部通过，src/（除legacy）LE操作0命中

### 3.3 DEBT-H-005-集群: 内存优化（P3/P4）

**问题**: 4项内存优化债务未清偿（LRU/Pool/懒加载/压缩）

**方案**: 存储层全面重构

| 模块 | 功能 | 性能指标 | 状态 |
|------|------|----------|------|
| lru-cache.ts | LRU淘汰策略 | 命中率95% | ✅ |
| buffer-pool-config.ts | 可配置Pool | 参数注入 | ✅ |
| lazy-loader.ts | 按需加载 | 延迟1ms | ✅ |
| zstd-compression.ts | 压缩存储 | 压缩率99.9% | ✅ |

**净债务变化**: -2（清偿4项，新增2项P4低风险）

### 3.4 DEBT-GOV-001: 债务预算制（P2）

**问题**: 缺乏债务管控机制，债务可能重新堆积

**方案**: 三层次治理体系
1. **PR层**: 强制债务声明模板
2. **CI层**: 自动检查脚本，预算超限（P1>2或P2>5）失败
3. **看板层**: DEBT-CURRENT.md自动更新

**预算规则**:
```yaml
P1: ≤ 2项（阻塞级）
P2: ≤ 5项（高优先级）
P3: ≤ 10项（中优先级，建议）
P4: 无限制（低优先级，跟踪）
```

**当前预算状态**: P1=0, P2=0, P3=2, P4=4 ✅ 健康
